## 一、为什么需要一体化的物联网模块

物联网项目常见痛点是「工具链割裂」：设备注册在一个系统，告警在另一个脚本里查邮件，业务系统又要单独对接 MQTT 或私有协议。Skylark 将设备模型、接入配置、出站订阅与告警通知放在同一权限与租户体系下，减少重复对接与运维成本，适合希望**快速上线、可持续迭代**的企业团队。

---

## 二、设备管理：从「产品」到「单台设备」

### 1. 产品与物模型

- **产品**对应一类设备形态，支持新建、编辑、启用/禁用、复制等日常运维操作。  
- **物模型**定义属性、事件、服务等语义，让上行数据与业务含义对齐，便于后续告警条件、订阅筛选与展示。

### 2. 设备与运行数据

- 在产线下维护**设备实例**，支持启用/禁用、密钥重置等安全操作，并提供**批量导入**，适合大批量落地。  
- 可查看**当前属性**与历史记录：属性上报、事件、服务调用、连接记录等，便于排查「设备到底有没有上报」「最近一次在线是什么时候」。

### 3. 设备分组

- 支持按业务维度建立**设备分组**，并在分组内维护设备成员，便于按区域、项目或客户维度做治理，也为告警与订阅按范围筛选打下基础。

---

## 三、接入与上行：对接主流消息中间件

平台提供面向接入侧的 HTTP 能力（如与 **EMQX** 等 Broker 协同的认证、Webhook 等配置），将设备连接与消息上行纳入统一配置；上行数据经处理后与产品物模型对齐，进入管理与使能模块。  
（具体部署依赖您的网络与 Broker 版本，实施阶段可按官方手册完成对接与联调。）

---

## 四、应用使能：把物联网数据送进您的业务系统

### 1. 数据订阅（出站集成）

面向「要把设备事件/属性变化推给我方 API」的场景：配置**出站通道**与**订阅规则**，按事件类型、产品、设备等条件过滤，将标准化后的数据推送到您的 HTTP 服务，并保留**投递记录**；失败支持**重试策略**，降低因网络抖动导致的丢数焦虑。

### 2. 消息订阅（通知）

面向「人要第一时间知道」的场景：与数据订阅类似，可配置**通知通道**与**订阅**，在告警或关键事件发生时，通过多种通道送达运维或业务人员。

当前通道类型包括但不限于：

- **邮件（SMTP）**  
- **钉钉 / 企业微信 / 飞书**（机器人 Webhook）  
- **短信**：可按项目扩展对接具体短信网关（平台侧预留扩展点）

消息支持**模板渲染**，并记录每次投递结果；失败同样可**手动或自动重试**，便于审计与故障追溯。

### 3. 告警管理

- **告警规则**：基于设备与分组等维度配置规则，与实时数据评估联动。  
- **告警记录**：统一查询历史告警，便于复盘与统计。  
- 告警产生后，可与上述**消息订阅**衔接，形成「检测 → 记录 → 通知」的闭环。

## 五、企业级能力：权限、租户与可运维性

- **权限控制**：控制台菜单与 API 按权限标签细分到查看、编辑、删除等操作，便于按角色授权（如仅运维可看设备密钥，业务同学仅看报表类能力）。  
- **多租户与数据域**：核心数据表支持租户隔离与审计字段，适合 SaaS 或多组织共用一套平台的部署方式。  
- **可观测性**：健康检查等运维端点便于接入现有监控体系。

## 六、适合谁使用

- 正在建设 **设备接入 + 后台运营** 一体化能力的制造业、能源、楼宇、农业等团队。  
- 已有业务系统，希望通过 **HTTP 订阅** 消费设备侧数据，而不希望从零维护 MQTT 消费与规则引擎的团队。  
- 需要 **告警到人**（邮件、协作工具机器人等）且希望投递过程**可追溯、可重试**的运维场景。

---

## 七、结语

Skylark 物联网模块的定位不是「又一个单一功能的设备台账」，而是覆盖**建模—接入—治理—出站—告警—通知**的一条链路。若您正在评估物联网中台或设备管理平台，欢迎结合自身场景拆分模块试用：先做产品与设备治理，再逐步打开数据订阅与告警通知，通常能以较小步长验证价值。
